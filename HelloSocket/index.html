  • 关于Windows

    Windows 下的 socket 程序和 Linux 思路相同，但细节有所差别：
    1) Windows 下的 socket 程序依赖 Winsock.dll 或 ws2_32.dll，必须提前加载。DLL 有两种加载方式，请查看：动态链接库DLL的加载

    2) Linux 使用“文件描述符”的概念，而 Windows 使用“文件句柄”的概念；Linux 不区分 socket 文件和普通文件，而 Windows 区分；Linux 下 socket() 函数的返回值为 int 类型，而 Windows 下为 SOCKET 类型，也就是句柄。

    3) Linux 下使用 read() / write() 函数读写，而 Windows 下使用 recv() / send() 函数发送和接收。

    4) 关闭 socket 时，Linux 使用 close() 函数，而 Windows 使用 closesocket() 函数。

基础

socket()

int socket(int af, int type, int protocol);

  • af:address family. ipv4:AF_INET4 ipv6:AF_INET6

  • type:数据传输方式

    • SOCK_STREAM 表示面向连接的数据传输方式。数据可以准确无误地到达另一台计算机，如果损坏或丢失，可以重新发送，但效率相对较慢。常见的 http 协议就使用 SOCK_STREAM 传输数据，因为要确保数据的正确性，否则网页不能正常解析.
    • SOCK_DGRAM 表示无连接的数据传输方式。计算机只管传输数据，不作数据校验，如果数据在传输中损坏，或者没有到达另一台计算机，是没有办法补救的。也就是说，数据错了就错了，无法重传。因为 SOCK_DGRAM 所做的校验工作少，所以效率比 SOCK_STREAM 高。

    注意：SOCK_DGRAM 没有想象中的糟糕，不会频繁的丢失数据，数据错误只是小概率事件。

    有可能多种协议使用同一种数据传输方式，所以在 socket 编程中，需要同时指明数据传输方式和协议。

  • protocol:常用的有 IPPROTO_TCP 和 IPPTOTO_UDP 不用多介绍

一般情况下有了 af 和 type 两个参数就可以创建套接字了，可以将 protocol 的值设为 0，系统会自动推演出应该使用什么协议。

除非遇到这样的情况：有两种不同的协议支持同一种地址类型和数据传输类型。如果我们不指明使用哪种协议，操作系统是没办法自动推演的。

listen()和accept()

int listen(int sock, int backlog);  //Linux

listen()可以让sock进入被动监听状态，backlog为请求队列最大长度。

如果将 backlog 的值设置为 SOMAXCONN，就由系统来决定请求队列长度，这个值一般比较大，可能是几百，或者更多。当请求队列满时，就不再接收新的请求，对于 Linux，客户端会收到 ECONNREFUSED 错误，对于 Windows，客户端会收到 WSAECONNREFUSED 错误。

注意：listen() 只是让套接字处于监听状态，并没有接收请求。listen() 后面的代码会继续执行，直到遇到 accept()。accept() 会阻塞程序执行（后面代码不能被执行），直到有新的请求到来。

int accept(int sock, struct sockaddr *addr, socklen_t *addrlen);  //Linux

accept()返回一个新的socket,来和客户端通信,连接了参数中的sock和addr(存着客户端的IP和port)。

值得注意的是最后一个参数，addr的大小，要求传入的 是一个socklen_t型的指针，因此需要先声明出一个变量，传其地址进入。

阻塞模式

对于TCP套接字（默认情况下），当使用 write()/send() 发送数据时：

1) 首先会检查缓冲区，如果缓冲区的可用空间长度小于要发送的数据，那么 write()/send() 会被阻塞（暂停执行），直到缓冲区中的数据被发送到目标机器，腾出足够的空间，才唤醒 write()/send() 函数继续写入数据。

2) 如果TCP协议正在向网络发送数据，那么输出缓冲区会被锁定，不允许写入，write()/send() 也会被阻塞，直到数据发送完毕缓冲区解锁，write()/send() 才会被唤醒。

3) 如果要写入的数据大于缓冲区的最大长度，那么将分批写入。

4) 直到所有数据被写入缓冲区 write()/send() 才能返回。

当使用 read()/recv() 读取数据时：
1) 首先会检查缓冲区，如果缓冲区中有数据，那么就读取，否则函数会被阻塞，直到网络上有数据到来。

2) 如果要读取的数据长度小于缓冲区中的数据长度，那么就不能一次性将缓冲区中的所有数据读出，剩余数据将不断积压，直到有 read()/recv() 函数再次读取。

3) 直到读取到数据后 read()/recv() 函数才会返回，否则就一直被阻塞。

这就是TCP套接字的阻塞模式。所谓阻塞，就是上一步动作没有完成，下一步动作将暂停，直到上一步动作完成后才能继续，以保持同步性。

shutdown()

int shutdown(int sock, int howto);  //Linux

sock 为需要断开的套接字，howto 为断开方式。
howto 在 Linux 下有以下取值：

  • SHUT_RD：断开输入流。套接字无法接收数据（即使输入缓冲区收到数据也被抹去），无法调用输入相关函数。

  • SHUT_WR：断开输出流。套接字无法发送数据，但如果输出缓冲区中还有未传输的数据，则将传递到目标主机。

  • SHUT_RDWR：同时断开 I/O 流。相当于分两次调用 shutdown()，其中一次以 SHUT_RD 为参数，另一次以 SHUT_WR 为参数。

默认情况下，close()/closesocket() 会立即向网络中发送FIN包，不管输出缓冲区中是否还有数据，而shutdown() 会等输出缓冲区中的数据传输完毕再发送FIN包。也就意味着，调用 close()/closesocket() 将丢失输出缓冲区中的数据，而调用 shutdown() 不会。

UDP

无需经历连接过程，Server不必调用listen()和accept().Client无需connect().

TCP套接字知道目标地址信息。但UDP套接字不会保持连接状态，每次传输数据都要添加目标地址信息，

在某种意义上无法明确区分服务器端和客户端，只是因为其提供服务而称为服务器端，希望各位读者不要误解。

sendto(sock, msg, sizeof(msg), 0,  sockaddr*)&your_adr, sizeof(your_adr));
recvfrom(sock, msg, BUF_SIZE, 0, (sockaddr*)your_adr, &adr_sz);

第一次sendto时会向对方注册目标ip和port

每次UDP传输都是如下过程：1.注册 2.传 3.删除注册信息

这样，如果连续和一台机器通信。要多次进行1和3.浪费。故可以

connect(sock,(sockaddr*)&serv_adr,sizeof(serv_adr));
然后调用read 和 write 就可以了，不用每次都sendto和 recvfrom 了

epoll

传统的select/poll另一个致命弱点就是当你拥有一个很大的socket集合，不过由于网络延时，任一时间只有部分的socket是“活跃”的，但是select/poll每次调用都会线性扫描全部的集合，导致效率呈现线性下降。但是epoll不存在这个问题，它只会对“活跃”的socket进行操作—这是因为在内核实现中epoll是根据每个fd上面的callback函数实现的。那么，只有“活跃”的socket才会主动的去调用 callback函数，其他idle状态socket则不会。

当我们调用epoll_ctl往里塞入百万个句柄时，epoll_wait仍然可以飞快的返回，并有效的将发生事件的句柄给我们用户。这是由于我们在调用epoll_create时，内核除了帮我们在epoll文件系统里建了个file结点，在内核cache里建了个红黑树用于存储以后epoll_ctl传来的socket外，还会再建立一个list链表，用于存储准备就绪的事件，当epoll_wait调用时，仅仅观察这个list链表里有没有数据即可。有数据就返回，没有数据就sleep，等到timeout时间到后即使链表没数据也返回。所以，epoll_wait非常高效。

epoll_creat();

  • op：操作

    • EPOLL_CTL_DEL 删

    struct epoll_event{
    __uint31_t events;
    epoll_data_t data;
}

    其中第二个成员又是个联合体：

    typedef union epoll_data{
    void * ptr;
    int fd;
    __uint32_t u32;
    __uint64_t u64;

    看看设置的例子吧：

    event.events = EPOLLIN; //发送读取时
event.data.fd = sockfd;

    其中还可以有以下事件：

    • EPOLLIN 读
    • EPOLLOUT 输出缓冲为空，可以立即发送
    • EPOLLPRI 收到OOB数据
    • EPOLLRDHUP 断开连接或半关闭，在边缘触发方式下非常有用
    • EPOLLERR 错误
    • EPOLLET 以边缘触发方式得到事件通知
    • EPOLLONESHOT 发生一次事件后，相应文件描述符不再收到事件通知。因此需要向epoll_ctl的第二个参数传EPOLL_CTL_MOD,再次设置事件。

僵尸进程

wait用于销毁僵尸进程

  • 僵尸进程： 如果父进程未主动要求获得子进程的结束状态值，操作系统会一直保存，并让子进程长时间处于僵尸进程状态。==》父亲要负责回收孩子

调用wait时，程序将阻塞，直到有子进程终止。因此需谨慎。

waitpid，不阻塞

pid_t waitpid(pid_t pid, int* statloc, int options); 返回终止的子进程id 失败返回-1

使用:

while(!waitpid(-1, &status, WNOHANG)){
    //xxx;
}
  • -1的意思是任意子进程
  • 第二个参数，接收返回值
  • WNOHANG：即使没有终止的子进程，也不阻塞，而是返回0并退出

信号

signal(SIGALRM, func1); //配合alarm(5) 5秒后唤起
signal(SIGINT, func2); // 输入Ctrl+C
signal(SIGCHLD, func3); // 子进程终止

signal完全可以被下面sigaction的替代。使用signal只是为了兼容旧程序。

sigaction三个参数

  • 要调用的函数，函数参数会传入唤醒的几种方法比如上面的三种
  • 一个叫sigaction的结构体地址
    • sa_handler 要调用的函数
    • 后面两个成员设为0即可(line 15-16)
  • 获取之前注册的信号处理函数指针，不需要则传0
#include <stdio.h>
#include <unistd.h>
#include <signal.h>
void timeout(int sig)
{
    if(sig==SIGALRM)
        puts("Time out!");
    alarm(2);    
}
int main(int argc, char *argv[])
{
    int i;
    struct sigaction act;
    act.sa_handler=timeout;
    sigemptyset(&act.sa_mask);
    act.sa_flags=0;
    sigaction(SIGALRM, &act, 0);
    alarm(2);
    for(i=0; i<3; i++)
    {
        puts("wait...");
        sleep(100);
    }
    return 0;
}

进程间通信

  • 管道

    速度慢 容量有限 只有父子进程能通讯

  • FIFO 有名管道

    任何进程间都能通讯 但速度慢

  • 消息队列、

    容量受系统限制 第一次读时要考虑上一次没有读完的数据

  • 信号量

    不能传递复杂消息 只能用来同步

    容易控制容量 速度快 但要保持同步（安全问题

锁

悲观锁

不加锁就出事！

进入一个同步、线程安全的方法时，要先获得这个方法的锁。退出这个方法时，则释放锁。

  • 重量级锁

    重量级锁要点：获取不到锁 马上阻塞

    线程阻塞时，不仅要保存线程此时的执行状态、上下文数据，还涉及用户态到内核态的转换，阻塞唤醒也非常消耗时间。

  • 自旋锁

    不立马阻塞 等一段时间

    设置循环等待几次 还拿不到锁就阻塞

  • 自适应自旋锁

    根据最近锁的状态调整上述自旋锁循环次数。而不是循环固定次数。

    如有的锁很少拿到，则少循环几次。因为多做几次拿到的概率也不大。有的经常拿到，就多循环几次可能就拿到了。

乐观锁

不加锁也没事，出了事再想办法解决

加锁需要通过操作系统。悲观锁每次都要加锁放锁。如果根本没有线程来竞争，白白上锁。

  • 轻量级锁

    用一个变量来记录此刻是否有人在执行。比起加锁，采用CAS改变状态，花销小多了。

    CAS可以保证操作的原子性

    适用于多个进程总是错开

  • 偏向锁

    更省事。首次进入函数时，采用CAS机制标记为有人在用(会记录线程id)。退出时不再改了。（偏向锁认为不会有其他进程来竞争）

    再次进入函数时，一看id是自己。继续执行。

    但如果id不是自己怎么办呢。升级为轻量级锁。
